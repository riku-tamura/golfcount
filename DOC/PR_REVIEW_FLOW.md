# PR Review Flow

## 目的

PR ごとにレビューの流れを固定し、指摘、修正対応、修正内容コメント、再レビュー、承認、マージまでを毎回同じ手順で進めるためのドキュメントです。

## 基本フロー

1. 作業ブランチを push して PR を作成する
2. `gh pr comment` を使って `riku-tamura` アカウントで `@codex日本語でレビューしてください` を投稿する
3. `chatgpt-codex-connector` から返信コメントまたはレビューが返るまで待機する
4. PR 作成後に必ずレビューを行う
5. 問題があれば、レビューコメントとして指摘を残す
6. 修正が必要なら、実際にコードや設定やドキュメントの修正対応を行う
7. 修正後に、何を直したかをコメントで残す
8. 修正しない判断なら、その理由をコメントで残す
9. 指摘対応後に、あらためてレビューを行う
10. 問題がなくなったら `LGTM` コメントを残す
11. 別レビュワーがいる場合は `approve` を付ける
12. その後に merge を行う

PR に新しい commit を push した場合も、同じ手順で `@codex日本語でレビューしてください` を再度コメントし、返信が返るまで待機してから次のレビューに進みます。

## コメントの残し方

### 指摘がある場合

- レビュワーとしてコメントを残す
- 内容は「何が問題か」「なぜ問題か」「どう直すとよいか」が分かるようにする

例:

- `この処理だとホール切り替え時に前の値が残る可能性があります。初期化位置を見直したほうがよいです。`

### 修正した場合

- まず実際の修正対応を行う
- 修正後にコメントを残す
- どの指摘に対して、どう直したかを明記する

例:

- `対応しました。ホール切り替え時に対象ホールの状態を再読込するように変更しています。`

### 修正しない場合

- 修正しない理由をコメントで残す
- 判断理由があとから追える状態にする

例:

- `今回は対応しません。現在の仕様ではこの動作を意図しており、別タスクで見直す前提のためです。`

### 問題がなくなった場合

- 最終レビューで `LGTM` コメントを残す
- 別レビュワーがいる場合はそのあとに `approve` を行う

例:

- `LGTM`

## Codex レビュー依頼の前提

- `chatgpt-codex-connector` GitHub App が対象リポジトリに接続されていること
- Codex 側で GitHub レビューが利用可能なプラン / 設定になっていること
- レビュー依頼コメントは `gh pr comment` で `riku-tamura` アカウントから投稿する
- コメント内容は `@codex日本語でレビューしてください` で統一する
- Draft PR の間は依頼せず、`Ready for review` 後に投稿する

## merge の条件

- 指摘事項に対する対応方針がコメントで残っていること
- 修正が必要な指摘は、実際の修正対応が反映されていること
- `@codex日本語でレビューしてください` の依頼後に、返信コメントまたはレビューが返ってきていること
- 再レビューが完了していること
- 問題がないと判断できること
- `LGTM` コメントが完了していること
- 別レビュワーがいる場合は `approve` が完了していること
- 必要な CI が成功していること

## このリポジトリでの運用方針

- `main` には直接コミットしない
- すべて PR ベースで反映する
- 原則 `squash merge` を使う
- merge 後はブランチを削除する
- 1人運用で自分が PR 作成者の場合は、自分自身への `approve` はできないため `LGTM` コメントまでを完了条件にする
